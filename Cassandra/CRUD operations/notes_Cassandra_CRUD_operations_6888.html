Notes: Cassandra- CRUD operations

CRUD operations
CRUD stands for Create, Read, Update, Delete. These are basic operations performed on any table in RDBMS.
Before creating any table, one needs to create a keyspace in Cassandra which holds the table.

In Apache Cassandra, a keyspace is the top-level container for your data, analogous to a database in relational databases. It acts as a logical grouping for tables, defining how data is replicated and distributed across the cluster. Keyspaces also control data replication and durable writes. 
Keyspaces in Cassandra
Data organization    
Keyspaces are used to group related tables together, providing a namespace for data within a Cassandra cluster. 
Replication 
Each keyspace has a replication strategy that determines how data is replicated across the cluster, influencing data consistency, availability, and performance. 
Durable writes 
Keyspaces can be configured to enable or disable durable writes, which affect the reliability of data writes. 
Analogous to RDBMS 
Keyspaces in Cassandra are similar to databases in relational database management systems (RDBMS). 
Replication maps 
Keyspaces utilize replication maps to define how data is replicated in different data centers within the cluster. 
User-defined types 
Keyspaces can also contain user-defined types, allowing for the creation of custom data structures. 
Single keyspace per application 
It's common practice to have one keyspace per application within a Cassandra cluster.    
Creating the KEYSPACE
CREATE KEYSPACE <identifier> WITH <properties>
The above statement creates a keyspace.
e.g
CREATE KEYSPACE KeySpace Name
WITH replication = {'class': Strategy name, 'replication_factor' : No.Of   replicas};
or
CREATE KEYSPACE KeySpace Name
WITH replication = {'class': Strategy name, 'replication_factor' : No.Of  replicas}
AND durable_writes = Boolean value;    
Understanding the terms
Replication
The replication option is to specify the Replica Placement strategy and the number of replicas wanted. The following table lists all the replica placement strategies.

Strategy name  
There are mainly 3 options for strategy names.
1Simple Strategy Specifies a simple replication factor for the cluster.
2Network Topology StrategyUsing this option, you can set the replication factor for each data-center independently.
3Old Network Topology Strategy This is a legacy replication strategy.
Using these options, one can instruct Cassandra whether to use commit log for updates on the current KeySpace. This option is not mandatory and by default, it is set to true.

Example
Given below is an example of creating a KeySpace.
Here we are creating a KeySpace named KnowIT.

We are using the first replica placement strategy, i.e.., Simple Strategy.
And we are choosing the replication factor to 3 replica.

cqlsh> CREATE KEYSPACE KnowIT
             WITH replication = {'class':'SimpleStrategy', 'replication_factor' : 3};

Durable_writes
By default, the durable_writes properties of a table is set to true, however it can be set to false. You cannot set this property to simplex strategy.

Example
Given below is the example demonstrating the usage of durable writes property.

cqlsh> CREATE KEYSPACE test
             WITH REPLICATION = { 'class' : 'NetworkTopologyStrategy', 'datacenter1' : 3 }
             AND DURABLE_WRITES = false;    

Verification    
One can verify whether the keyspace  is created or not using the command Describe. If you use this command over keyspaces, it will display all the keyspaces created as shown below.

cqlsh> DESCRIBE keyspaces;    

cqlsh> SELECT * FROM system_schema.keyspaces;
Above command  verifies whether the durable_writes property of test KeySpace was set to false by querying the System Keyspace. This query gives you all the KeySpaces along with their properties.

update data
As like RDBMS we can update the data stored in the table using update clause.
Cassandra updates data only on primary key. Hence a table must have PK if update and delete transactions needs to be performed.
Syntax
UPDATE table_name
SET column_name= new value;
e.g   update emp
set deptno=20
where empno=7875;       
Adding new element in the  array
Update hobbies of Ranjan and add new hobby swimming and gaming.
update emp
set hobbies=hobbies+{'swimming','gaming'}     
where roll=123;
here, roll has to be PK.
