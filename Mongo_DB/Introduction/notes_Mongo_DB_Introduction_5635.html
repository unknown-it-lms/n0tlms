Databases are widely divided into two major types or categories, namely, Relational or Sequence Databases and Non-relational or Non-sequence databases or No SQL databases. An organization may use them individually or combined, depending on the nature of data and functionality required.
NoSQL Database (not only SQL)

The main types are document, key-value, wide-column, and graph. They provide flexible schemas and scale easily with large amounts of data and high user loads.

Industry at this phase needed data to be handled without any boundries. Social media needed to store the data which had no definite format. Hence MongoDB was developed and now is one of the very popular database.

About MongoDB
Primary datastore used by web developers is a relational database. This database is a structured database and the data  maintained in this structured database is a normalized data. Structured database dose not provide the flexibility to store required data. (e.g if multiple  email ids of employee are maintained in the database, such provision is to be made before hand while creating the table. To store variable number of email ids in structured database is a impossible task).   
Hence the concept of unstructured database came up.  In this case data is not-normalized. This type of datastore is more appropriate for internet data, where following a fixed structure is  a challange for developers.
MongoDB is an ideal example of unstructured  database.

Somewhere in mid-2007, 10gen, a startup in NewYork began work on PaaS(Platform-as-a-Service), composed of an application server  and  database, that would host web applications and  could scale as per the need. Like Google's App Engine, 10gen's platform was designed to handle the scaling and management of hardware and software infrastructure automatically, freeing developers to focus solely on their application code.  Here 10gen ultimately found out that most developers were not comfortable in giving up so much control over their technology stacks, but users always wanted 10gen's latest technology. This led 10gen to concentrate its efforts on database. This database became MongoDB. 10gen since then changed its name as MongoDB Inc. MongoDB is developed  by Dwight Merriman, Eliot Horowitz and Kevin Ryan.

At the Internet advertising company DoubleClick (now owned by Google), the team developed and used many custom data stores to work around the shortcomings of existing databases. The business served 400,000 advertises per second, but often struggled with both scalability and agility. Frustrated, the team was inspired to create a database that tackled the challenges it faced at DoubleClick. This was when MongoDB was born.  Since than MongoDB is widly used as a datastore for social media platforms.   

MongoDB is a document database built for general-purpose usage. It stores data in an optimized JSON format called BSON (Binary JSON). This allows the MongoDB document model to benefit from the flexibility of JSON documents stored in logical groups called collections. The document model supports many different data types, including strings, dates, numbers, arrays, decimals, nested objects, geo data, and binary data.
MongoDB is a document database built for general-purpose usage. It stores data in an optimized JSON format called BSON (Binary JSON). This allows the MongoDB document model to benefit from the flexibility of JSON documents stored in logical groups called collections. 

MongoDB uses replication and data partitioning called sharding to distribute data for high availability and scalability purposes, making it a highly consistent and fault tolerant database. MongoDB supports multiple deployment methods, from running it for free on a variety of platforms and servers starting from your local machine to a fully blown deployment in the cloud of your choice using MongoDB Atlas. Additionally, MongoDB Enterprise Advanced and MongoDB Atlas offer enterprise-grade security features like authentication, authorization, and LDAP support as well as end-to-end encryption.

MongoDB’s unified query API and powerful aggregations, in conjunction with the flexibility of the document model, has made it the most popular general-purpose document database in  the market.

MongoDB is classified as a document-oriented NoSQL database . Here are some key characteristics that define MongoDB .

Key Characteristics of MongoDB
1.  Document Store 
     MongoDB stores data in documents, which are similar to JSON objects. A document is represented in BSON      (Binary JSON) format, allowing for rich data types, including arrays and nested documents.

2.  Flexible Schema 
     Unlike traditional relational databases, MongoDB uses a flexible schema. This means documents within the same collection can have different structures, allowing for greater adaptability as application requirements evolve.

3. Collections 
    Documents are grouped into collections, analogous to tables in relational databases. Each collection can hold documents of varying shapes and sizes.

4.  Scalability 
     MongoDB is designed to scale horizontally. It supports sharding, which allows data to be distributed across multiple servers for better performance and availability.

5.  Rich Query Language      
     MongoDB supports a rich query language that allows for complex queries, including filtering, sorting, and aggregating data. It also supports indexing to improve query performance.

6.  Full-Text Search and Geospatial Queries 
     MongoDB includes features that support full-text search and geospatial queries, making it suitable for a wide range of applications.

7.  Data Relationships 
     While MongoDB is document-oriented, it provides tools for managing relationships between data, such as embedding documents or using references.

MongoDB is commonly used in scenarios where the following conditions apply 
- Application data is unstructured or semi-structured.
- Rapidly changing requirements arises the need of flexible schemas.
- There is a need for handling large volume of data with high write and read loads.
- Real-time analytics and big data applications.

Overall, MongoDB's document-oriented model provides a high degree of flexibility and scalability, making it a popular choice for modern application development, especially in environments that require rapid iteration and evolution.
Mongo Internals
Data model of MongoDB is Document orianted.  Record/row in RDBMS is refered as a document and a table  is refered as a Collection.  These Collections are containers for structurally or conceptually similar documents.

In Oracle /MySQL , data is stored in table format.  MongoDB use JSON (Java Script Object Notation) format to store data. To be more specific the data in MongoDB uses BSON (Binary Script Object Notation) i.e all data is stored in binary format. 

MongoDB uses replication technique to store the data. Data replication is the process of creating and maintaining multiple copies of the same data in different locations as a way of ensuring data availability, reliability and resilience across an organization. By replicating data from a source location to one or more target locations,  organization’s global users get ready access to the data they need without suffering from latency issues.
     
When multiple copies of the same data exist in different locations, even if one copy becomes inaccessible due to disaster, outage or any other reason, another copy can be used as a backup. This redundancy helps organizations to minimize downtime and data loss and improve business continuity

Replication exists primarily to offer data redundancy and high availability. We maintain the durability of data by keeping multiple copies or replicas of that data on physically isolated servers. The process of creating redundant data to streamline and safeguard data availability and durability is called as data replication.
Multiple copies of data are maintained across servers. This is especially useful if a server crashes or if you experience service interruptions or hardware failure.

If your data only resides in a single database, any of these events would make accessing the data impossible. But  replication, your applications can stay online in case of database server failure, while also providing disaster recovery and backup options.

With MongoDB, replication is achieved through a replica set. Writer operations are sent to the primary server (node), which applies the operations across secondary servers, replicating the data.

If the primary server fails (through a crash or system failure), one of the secondary servers takes over and becomes the new primary node via election. If that server comes back online, it becomes a secondary once it fully recovers, aiding the new primary node.

 If the primary server fails a secondary server can take over soon. This process is intended to be mostly transparent to the user, with drivers taking care of routing queries to the requested replica. Replication in MongoDB is facilitated through replica sets.       

Master-slave(Client Server) replication is a technique in MongoDB that ensures data is always available to users, even if one system fails.
Primary node -  Formerly known as the "master", this node receives all write operations.
Secondary nodes - Formerly known as the "slaves", these nodes replicate data from the primary node.
Automatic failover - If the primary node fails, one of the secondary nodes is automatically elected as the new primary.

Sharding   
Replication technique is used in MongoDB, which creates multiple copies of data on the servers. These replication  servers are called as Shards.
Database sharding is a database management technique that involves partitioning a growing database horizontally into smaller, more manageable units known as shards.

Sharding is especially useful in cases where you’re working with large amount of data, as it allows you to scale your base horizontally by adding more machines that can function as new shards.

Generally, a MongoDB database contains a large number of collections. Every collection consists of various documents that contain data. You can break up this large collection into multiple smaller collections using sharding. This allows MongoDB to perform queries without putting much strain on the server.

  ​

Difference between replication and sharding 
Replication -  The primary server node copies data onto secondary server nodes. This can help increase data availability and act as a backup, in case if the primary server fails.

Sharding - Handles horizontal scaling across servers using a shard key. This means that rather than copying data holistically, sharding copies pieces of the data (or “shards”) across multiple replica sets. These replica sets work together to utilize all of the data.   

Replication and sharding can work together to form  sharded cluster, where each shard is replicated in turn to preserve the same high availability.
