Notes: OOP_Using_Java- Exception Handling

Exception Handling
A programmer is mainly responsible for following two aspects in a software.
  • Syntax of the language
  • Logic of the program.

However, success or failure of a software depends on many other factors which are beyond the control of the programmer. For example:
  • Memory available on the system 
  • Availability of a File needed by the program.
  • Connectivity of Database server our program is trying to connect
  • Availability of internet connectivity needed by our system
  • Correct value and format of the data accepted from the user by our program
  • Many other related to external resources etc

Errors related to above mentioned scenarios occur only when the program is running in a given context. Hence they are called as run-time error or exceptions. Such errors put our program in such a state that further execution becomes invalid. For example, if the file we are tying to open is not available, there is not point in proceeding further. An alternate path of execution should be provided for our system to come out of this state. Exception handling mechanism handle such run-time error situations without disturbing the code written for normal execution.

Lets see some common scenarios that may fail at the time of execution.
Person p1= new Person();We are trying to create a new object of class Person here. This may fail if there is not enough memory available for the program at the time of execution of this statement. 
/* Value of a and b is accepted from the user */
double c= a/b;		Here, we are dividing number a by number b whose values are accepted from the user. If user enters 0 for b, then this leads to division by zero which is an error during the time of such execution. 
/* Value for str is accepted from the user */
int a= Integer.parseInt(str);		If the user enter a wrong value (like "12D" ) for str, which cant be parsed as an integer, this statement will give an error.
File file = new File("D:/know-it/prn.txt");We are trying to open a file from a given location. If this file is not present at the given location with given name, this statement will give error at the time of execution.

Why throwing exceptions is better than returning a value that indicates error.
  1. A function may return some value that indicates result of exception of that function. In such case, we cant return the error value due to following reasons:
    a) A function can't return more than one values.
    b) Even if we decide to return the error value, in some situations, we may not be able to differentiate     between the success value and error value.
  2. There are some functions in OOP, like constructors, which don't return anything. We can't use the return statement in such functions.
Hence, exceptions are better mechanism for handling such error scenarios.

Try throw catch
Java uses try, throw and catch for basic implementation of exception handling. 
try {

    // If some error occurs here..
    throw new Exception();
}
catch(Exception ex) {
   
}		/* try is used to mark a block of code that has possibility of going wrong at run-time due to reasons beyond the control of the programmer.*/
/* throw is used to actually create the exception that encapsulates necessary details of the error scenario. This exception object will be thrown from the point of failure. */
/* catch is used to mark a block of code that handles the exception and tries to take some corrective action.
*/
The exception can be thrown by 
  1. The code written in try block directly OR
  2. By another user defined function called from try block OR
  3. By some built-in function called from try block OR
  4. By JVM due to some invalid operation

Note that, we can even write one catch block for more than one type of exceptions, as shown below:
public class Demo {

            // This catch block will handle all exceptions of remaining types
        }
    }
}

Exception hierarchy
Exceptions in Java are classified as below.
​
Important points to note:
  1. All exceptions are extended from class Throwable
  2. Throwable has two sub-classes, Exception and Error. Exceptions are the run-time issues which are expected to be handled by the programmer. Errors are the issues which are beyond the control of the programmer. (For example, the ArrayOutOfBoundException is a runtime exception, which the programmer can avoid by checking the array bounds. NullPointerException is a runtime exception which the programmer can avoid by checking if the reference is not null before using it further. However, OutOfMemoryError indicates that the machine has no memory left anymore, for which the programmer can't do anything except reporting the issue in logs.)
  3. There are two types of exceptions.
    • Checked exceptions
    • Un-Checked exceptions
  4. Checked exceptions are extended from Exception class
  5. Un-Checked exceptions are extended from Run-time exception or Error class
Checked- unchecked exception
Exceptions in Java are of two types, checked and un-checked. We will see following points in this topic:
  1. Checked exceptions
  2. Un-checked exceptions
  3. Rules of exceptions in function overriding
1. Checked exceptions
These are the exceptions which must either be caught by the code that generates it OR must be declared using the throws keyword. If a code fails to take any of these two actions, the compiler gives error during compilation. This is Handle or Declare rule.
Exceptions that are derived from class Exception directly excluding RuntimeException and its sub-classes are checked exceptions.

Lets understand this using an example.
public static void Read()
{
    FileReader file = new FileReader("C:\\test\\mydata.txt"); /*Error: Un-handled exception type FileNotFoundException */
    BufferedReader fileInput = new BufferedReader(file);
    String str= fileInput.readLine(); /*Unhandled exception type IOException */
    fileInput.close(); /*Un-handled exception type IOException */
}
Note that: 
  • The constructor of FileReader, throws a checked exception of type FileNotFoundException. As this exception is neither handled in the code nor declared using the throws clause, compiler will give error on that line.
  • The functions readLine() and close() throw another checked exception of type IOException. Even this exception is neither handled in the code nor declared using the throws clause. Hence we get compile-time error even on these two lines.
We can solve this error in following different ways:
1. By handling all exceptions in the code that generates them, as shown below:
public static void Read() 
{
    try
    {
        FileReader file = new FileReader("C:\\test\\mydata.txt");
        BufferedReader fileInput = new BufferedReader(file);
        String str= fileInput.readLine();
        fileInput.close();
    }
    catch(FileNotFoundException ex) {
    }
    catch(IOException ex) {
    }
}
  • Here, all types of checked exceptions are handled using try-catch block inside the function Read() itself. So there is no checked exception that goes un-handled in the Read() function. So there is not compile-time error.
  • Here, we can even remove the catch block with FileNotFoundException parameter, because the other catch block with IOException, can handle both types of exceptions.  
2. By declaring all exceptions using throws clause, as shown below:
public static void Read() throws FileNotFoundException, IOException
{
        FileReader file = new FileReader("C:\\test\\mydata.txt");
        BufferedReader fileInput = new BufferedReader(file);
        String str= fileInput.readLine();
        fileInput.close();
}
OR
public static void Read() throws IOException
{
        FileReader file = new FileReader("C:\\test\\mydata.txt");
        BufferedReader fileInput = new BufferedReader(file);
        String str= fileInput.readLine();
        fileInput.close();
}
  • Here, we have not handled the checked exceptions thrown inside the function Read() BUT we have declared both of them using the throws clause.
  • Note that, here we can even declare only IOException in the throws clause, because IOException is the super class of FileNotFoundException. FileNotFoundException is-a IOException. So there is no need to declare it seperately. 
2. Un-checked exceptions
These are the exceptions which are not checked by the compiler. So even if we don't handle them using try-catch at the code that generates them OR don't declare it using throws clause, the compiler won't give any error.
Lets understand this using following code:
public static void Add()
{
    int iTotal=0;
    int arr[]= new int[5];
    for(int i=0;i<=5;i++) { /* Here ArrayIndexOutOfBoundsException will be thrown */
        iTotal+= arr[i];
    }
    float average=iTotal/0; /* Here, as the denominator is zero, ArithmeticException will be thrown.*/
}		In this code, two types of exceptions will be thrown. But both are un-checked exceptions. Hence, it is not compulsory to catch them or even declare them using throws clause. 
Hence, though this code throws exception are run-time, it won't given compile-time error.
public static void Add() throws ArithmeticException, IndexOutOfBoundsException {
// ...
}		We can even declare un-checked exception using throws clause but it is NOT compulsory.
Exceptions that are derived from class RuntimeException and class Error are un-checked exceptions. Refer the Exception hierarchy given in previous point. The class with green background are Checked exceptions and remaining are Un-checked exceptions.

3. Rules of exceptions in function overriding
Following are the rules related to exceptions while overriding a method in sub-class.
  • If the method in super class  doesn't throw any exception:
    • Method in sub-class can NOT throw any checked exception
    • Can throw any un-checked exception
class Person {
    public void Display() {
    }
}
Here, Display method in super class Person does NOT throw any exception. So, Its overridden method in sub-class
  • Can throw un-checked exception
  • Can not thrown any checked exception  
class Employee extends Person{
    public void Display()
        throws ArithmeticException {
    }
}
/* This is allowed because ArithmeticException is un-checked exception*/
class Employee extends Person{
    public void Display()
        throws IOException {
    }
}
/* This is NOT allowed because IOException is checked exception and the super class method does NOT throw any exception*/
  • If the method in super class  throws an exception of type, say A:
    • Method in sub-class cant throw only checked exception of type A and other types extended from A 
    • Can throw any un-checked exception
    • Can decide not to throw any exception.
class Person{
    public void Display() throws IOException {
    }
}
Here, Display() in super class throws a checked exception, IOException. So, 
Overridden method Display() in sub-class 
  • Can decide not to throw any exception
  • Can throw any un-checked exception
  • Can throw exception of type IOException or any other exception extended from IOException (like FileNotFoundException)
  • Can NOT throw an un-checked exception which is not extended from IOException.
class Employee extends Person{
    public void Display() {
    }
}
/*This is OK. Method in sub-class not declaring any throws clause */
class Employee extends Person{
    public void Display() throws ArithmeticException {
    }
}
/*This is OK. Method in sub-class throwing un-checked exception*/
class Employee extends Person{
    public void Display() throws  FileNotFoundException {
        
    }
}
/*This is OK. Method in sub-class not declaring any throws clause*/
class Employee extends Person{
    public void Display() throws ClassNotFoundException{
        
    }
}
/*Error: Exception ClassNotFoundException is not compatible with throws clause in Person.Display()*/
In short, if the method of super class throws a checked exception of type A, method of sub class can NOT throw checked exceptions that are not of type A.  Everything else is allowed.

Custom exceptions
The Java exception hierarchy covers almost all scnearios which generally occur in all applicaiton developments. However, sometimes we may need to create our own exception class, to handle some scenario specific to our application. These are the scenarios that mainly handle the business logic related errors.

Following are some of such scenarions:
  • We may decide to throw an exception, say InvalidCredentialsException, if the username-password is invalid.
  • We may decide to throw an exception, say InvalidLicenseException, if the license of that software has expired or is invalid.
  • We may decide to throw an exception, say MaxAttemptsException, if the user tries to enter invalid password for more than 3 times. 
​​​​​​​
We need to create our own class for creating custome exception and extend it either from class Exception or class RuntimeException. If it is extended from class Exception, our custom exception becomes checked exception else it becomes un-checked exception. 
This is as shown below:
class FirstException extends Exception
{
    String strMessage;
    public MyException(String str) {
        strMessage= str;
    }
    public String toString() {
        return "MyException: "+strMessage;
    }
}
Here FirstException is a custom exception. It is checked exception because we have extended it from class Exception.
The Handle or Declare rule is now applicable to MyException also. This is as shown below.

public class Demo {
    public static void main(String[] args)
    throws FirstException {
        throw new FirstException();
    }
}
class SecondException extends RuntimeException
{
    String strMessage;
    public MyException(String str) {
        strMessage= str;
    }
    public String toString() {
        return "MyException: "+strMessage;
    }
}		Here SecondException is a custom exception. It is un-checked exception because we have extended it from class RuntimeException.
The Handle or Declare rule is NOT applicable to MyException also. Hence, it can be used as shown below.

public class Demo {
    public static void main(String[] args) {
        throw new SecondException();
    }
}
Note that, it is programmers decision to provide differnet types of constructors, getters, toString() etc functionality in the custome exception class to make it more programmer-friendly. It is not compulsory

​​​​​​​2. Rules of exceptions which belong to a hierarchy
A code can throw exceptions that belong to a hierarchy, as shown below. 
class A extends Exception {
}
class B extends A {
}
class C extends B {
}		/* These exceptions can be handle by a single catch block of most super class, as shown below */
public class Demo {
    public static void main(String[] args) {
        try
        {
            throw new B();
        }
        catch(A ex) {
    ​​​​​​​    /* Can handle A, B as well as C type of exceptions */
        }
    }
}
/* They can be handle by a separate catch block for each type, as shown below. BUT the order of catch should be from most sub-class to most-super class, as shown below */
public class Demo {
    public static void main(String[] args) {
        try
        {
            throw new B();
        }
        catch(C ex) { // Most-sub class first
        }
        catch(B ex) {
        }
        catch(A ex) { // Most-super class last
        }
    }
}		/* If we write a catch clock for super-class before that of a sub-class, it can handle sub as well as super type of exceptions and hence will make other catch block unreachable. This will give compile time error */
public class Demo {
    public static void main(String[] args) {
        try
        {
            throw new B();
        }
        catch(A ex) {
    /* This will handle A, B as well as C exceptions and hence will make next catch blocks unreachable. Hence, it gives error */
        }
        catch(B ex) {
        }
        catch(C ex) {
        }
    }
