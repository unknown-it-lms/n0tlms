Polymorphism
Object oriented programming supports two types of programmings:
  • Compile-time polymorphism (also called as false polymorpshim / static polymorpshim)
  • Run-time polymorphism (also called as true polymorpshim / dynamic polymorpshim)

Compile-time polymorphism
Function overloading and operator overloading are the examples of compile-time polymorpshim. They allow the programmer to write more than one functions with same name but with different signature. The compiler calls appropriate function based on the parameters passed while calling these functions.

Run-time polymorphism
This is the real polymorphism. We need to understand the concept of generic reference in Java to understand run-time polymorphism.
Generic reference
In Java, reference of super class can refer to objects of sub classes. Such type of reference is called as generic reference. 
Note that
  • Only base class reference can refer to objects of its own types and other types derived from it.
  • Derived class reference can't refer to objects of base class. It can't refer to objects of peer level classes also.
Employee e1, e2, e3,e4;

e1= new Employee("Sachin", 24, 4, 2001, 101, 21000); /* OK. Reference of class Employee e1, referring to object of same class Employee */

e2= new Programmer("Rahul", 23, 4, 1990, 101, 21000, ".NET Projects", 35, 1250); /* OK. Reference of class Employee, referring to object of its direct derived class Programmer */

Person p1= new Admin("Mahesh", 4, 5, 2001, 101, 21000, 12500); /* OK. Reference of class Person, referring to object of class Admin derived indirectly, in two levels of inheritance from Person*/

Programmer p1= new Employee("Sachin", 24, 4, 001, 101, 21000); /* ERROR. Reference of class Programmer, referring to object of its super class Employee*/

Programmer p2= new SalesManager("Manish", 5, 4, 1979, 101, 21000, 500000, 0.12); /* ERROR. Reference of class Programmer, referring to object of its peer level class SalesManager*/

Static and Dynamic binding
When ever a function is called in a program, the process of searching the exact function to be called, is called as function binding. 

There are two ways in which functions binding is done:
  1. Static binding
  2. Dynamic binding
Static bindingDynamic binding
In static binding, the decision of actual function to be called is taken by the compiler during compilation, hence it is called as compile-time binding or static binding. In Java, this is applicable for static functions which are called using a class name.In dynamic binding, the decision of actual function to be called is taken by the JVM during actual program execution, hence it is called as run-time binding or dynamic binding. In Java, this is applicable for all non-static member functions which are called using a class reference.

Implementing polymorphism
Polymorphism is different implementation of same request by different objects which are related to each other through IS-A type of relation. This is possible in Java due to the concepts of generic reference and dynamic binding.

Lets understand this through actual code.
​


Employee allEmp[]= new Employee[3]; /* This is an array of 3 Employee references */


// Here each element of this array is initialized with object of different sub classes of Employee


allEmp[0]= new Programmer("Rahul", 23, 4, 1990, 101, 21000, ".NET Projects", 35, 1250);


allEmp[1]= new SalesManager("Manish", 5, 4, 1979, 101, 21000, 500000, 0.12);


allEmp[2]= new Admin("Ajay", 10, 5, 1974, 101, 21000, 12500);

for(Employee e: allEmp) {

    System.out.println(e.display()); 
    /*
    Due to dynamic binding:
    For first iteration, reference e refers to Programmer object. So Programmer's display() gets called.
    For second iteration, reference e refers to SalesManager object. So SalesManager's display() gets called.
​​​​​​​    For third iteration, reference e refers to Admin object. So Admin's display() gets called.
    */
    /* Note that:
    The message is same [e.display()], but its implementation is different depending on the object receiving that message. Hence, it performs Polymprphism.
​​​​​​​    */
}

Typecasting
We have seen that generic reference can refer to objects of derived classes BUT there is a limitation to this. Using a generic reference, though we can refer to the object of derived class, we can't access the additional class members added by that derived class. We can only access those members which are already present in the class of generic reference.
If we want to access the additional members of the derived class, we have to type-cast the generic reference to the derived class type. After such type-casting, we can access the additional members of the derived class also. 
Such type of type-casting which changes a reference from super class to sub class is called as down-casting.

Lets understand this through the following example:
package Staff;

package test;
import staff.*;
import utility.*;

public class Test {
    public static void main(String[] args) {
    /* Note that Employee references e1, e2 and e3 can refer to objects of derived classes */    
    Employee e1= new Programmer("Rahul", 23, 4, 1990, 101, 21000,     ".NET Projects", 35, 1250);
    Employee e2= new SalesManager("Manish", 5, 4, 1979, 101, 21000,     500000, 0.12);
    Employee e3= new Admin("Ajay", 10, 5, 1974, 101, 21000, 12500);
    
    /* Generic reference can access members visible in its own class */
    int id1= e1.getEmpId(); // OK
    int id2= e2.getEmpId(); // OK
    int id3= e3.getEmpId(); // OK

    /* BUT they can't access the additional members that are added in the derived classes */

    String strProjectName=e1.getProjectName(); /* ERROR: The method getProjectName() is undefined for the type Employee */

    double target= e2.getTarget(); /* ERROR: The method getTarget() is undefined for the type Employee*/

    double allowance= e3.getAllowance(); /* ERROR: The method getAllowance() is undefined for the type Employee*/

/* This is possible after type-casting each reference to the derived class data type as follows */

    String strProjectName=( (Programmer) e1 ).getProjectName();
    double target= ( (SalesManager) e2 ).getTarget();
    double allowance= ( (Admin) e3 ).getAllowance();
    
    System.out.println("Project name: "+strProjectName);
    System.out.println("Target: "+target);
    System.out.println("Allowance: "+allowance);
    }
}

Static casting:
The typecasting done without verifying the type of object referred by the object is called as static casting. This should ONLY be used in those situations where we are sure about the data type of the object referred by the reference and validity of the downcasting.
To handle the above situation, in the function ShowDetails, we should not down-cast the reference e directly to SalesManager. We should do it ONLY after ensuring that the object referred by reference e is actually a SalesManager. This can be done using instanceof operator in Java as follows: 
  • instanceof operator return a boolean value (true or false).
  • It has two operands, first is a data type and second an object reference.
  • instanceof returns true if there is IS-A type of relation between the type of object (second operand ) and type of object referred by the reference (first operand) 


Employee e1= new Programmer("Rahul", 23, 4, 1990, 101, 21000, ".NET Projects", 35, 1250);

Employee e2= new SalesManager("Manish", 5, 4, 1979, 101, 21000, 500000, 0.12);

Employee e3= new Admin("Ajay", 10, 5, 1974, 101, 21000, 12500);

Boolean bResult;
bResult= e1 instanceof Programmer; 
System.out.println(bResult); /* true. e1 refers to Programmer object. Programmer IS-A programmer */

bResult= e1 instanceof SalesManager;
System.out.println(bResult); /* false. e1 refers to Programmer object. Programmer IS NOT A SalesManager*/

bResult= e1 instanceof Admin;
System.out.println(bResult); /* false. e1 refers to Programmer object. Programmer IS-NOT-A Admin*/

bResult= e2 instanceof SalesManager;
System.out.println(bResult); /* true. e2 refers to SalesManager object. SalesManager IS-A SalesManager*/

bResult= e2 instanceof Programmer;
System.out.println(bResult); /* false. e2 refers to SalesManager object. SalesManager IS-NOT-A programmer */

bResult= e2 instanceof Employee;
System.out.println(bResult); /* true. e2 refers to SalesManager object. SalesManager IS-A Employee*/
Using the instanceof operator, we can first check the validity of downcasting and do it only if it is valid. Else not.
This is called as conditional downcasting or safe downcasting.
public static void ShowDetails(Employee e) {
    String str=e.display(); /* display is applicable for all Employees because it is present in Employee class and even overridden in derived classes. So we can call it directly */
    System.out.println(str); 

    /* Now we do safe downcasting. Hence the code gets executed     only for valid downcasting */
    if(e instanceof SalesManager) {
        System.out.println("Target: "+((SalesManager) e).getTarget());
    }
}
Typecasting is of two types. 
  1. Upcasting 
  2. Downcasting

Lets see the differences between them.
Upcasting
Downcasting
Typecasting a derived class type to base class type is called as upcasting.
Typecasting a base class type to derived class type is called as downcasting.
Objects referred by derived class reference always have IS-A relation with base class. Hence, upcasting will always be successful.Objects referred by base class may or may not have IS-A relation with derived class. Hence, downcasting is not ALWAYS successful.
As upcasting is always successful, it can be done with static casting, without using instanceof operator.Downcasting must be done after checking its validity using instanceof operator.
