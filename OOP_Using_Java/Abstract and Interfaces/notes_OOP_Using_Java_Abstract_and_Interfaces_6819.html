Abstract and Interfaces
Many times we expect some behavior from an entity but can't define it meaningfully without knowing a more specialized form of it. For example, we know that every verhicle has a price. But we can't tell price of avehicle without knowing exact type of vehicle (bike or car or truck etc). Same is true for shapes. We know that every shape has length but we cant calculate its length without knowing specific type of shape (like line, triangle, rectangle etc).
Such functions, which are expected in the general entity but can be defined only in specific entities are abstract functions in object oriented programming.
Abstract function and classes
Functions which are expected in a class but can not be defined in that class are declared as abstract functions. These functions are defined in the derived class.

package Staff;
import Utility.*;

abstract public class Employee extends Person {
    int empId;
    float basicSal;
// Expected but can't be defined
  abstract public String getRole();
  }
Note that:
  • Every employee has a role in organization. So this behavior is expected from every employee. Hence, this function must be present in the class Employee.
  • But this function can't be defined without knowing specific type of the employee. Hence, it can't be defined in class Employee. Hence, such function is declared as abstract in class Employee. 
  • Now class Employee is incomplete because it has one undefined function. Hence, even the class must be declared as abstract.

Consequence/ Rules of declaring abstract function in a class:
  • The class having abstract class MUST be declared as abstract.

  • Objects of abstract class can not be created. 

  • However, its reference can be created. Its reference can be used to refer to objects of derived classes and invoke their functionalities.

  • If a class extends an abstract class and doesn't define the abstract method, even that class must be declared as abstract and it can't be instantiated. 
    ​​​​​​​
  • If a method is declared as abstract, it can not be defined.

final keyword
In Java, the final keyword is used for
  1. Variable
  2. Method
  3. Class
1. final variable
If a variable is declared as final, its value can not be changed.Final variable can be created in two ways:
  • Without initialization
  • With initialization
Final variable created without initialization
class A {

    final int a; // final without initialization

    public A() {
        a=10; // OK
        a=20; / ERROR: The final field a may already have been assigned */
    }
    public A(int a) {
        this.a=10; // OK
    }
    public void setA(int a) {
        this.a=a; /* ERROR: The final field A.a cannot be assigned */
    }
}
  • In this example, int a is declared as final but not initialized.
  • Such variables can be initialized ONLY in all the constructors.
  • Even in same constructor, they can't be reassigned a value if already assigned.
  • Their value can't be changed in any other functions
Final variable created with initialization
class A{
    final int a=10; // final with initialization
    public A() {
        a=10; // ERROR: A.a cannot be assigned
        a=20; // ERROR: A.a cannot be assigned   
    }
    public A(int a) {
        this.a=10; // ERROR: A.a cannot be assigned
    }
    public void setA(int a) {
        this.a=a; // ERROR: A.a cannot be assigned
    }
}
  • In this example, int a is declared as final and initialized with a value.
  • Such variables can NOT be initialized anywhere else. Not even in constructors.
class A {
    int a=20;
    public void setA(final int x) {
        this.a= this.a+x; // OK
        x++; // ERROR
    }
}
  • Even a function parameter (as well as local variable) can be declared as final.
  • In such cases, its value can be used but not changed.

2. final method
If a method is declared as final, it can not be overridden in derived classes.
class A {
    int a=10;
    /* Can't be override in derived classes */
    final public int getA() {
        return a;
    }
}

class B extends A { 

/*ERROR: Cannot override the final method from A */    public int getA() {
        return 20;
    }
}
  • In this example,getA() is declared as final method. Hence, it can't be overridden in derived class B.
  • However, the final method gets inherited in the derived class.
  • Final method MUST be defined.
  • We can't declare a final method as abstract.

3. final class
If a class is declared as final, it can not be extended further.
final class A {
    int a=10;
    final public int getA() {
        return a;
    }
}
class B extends A { /* ERROR: he type B cannot subclass the final class A */
}
  • If a class is declared as final, it can not be extended further.
abstract final class A { /*ERROR: The class A can be either abstract or final, not both */
    int a=10;
    final public int getA() {
        return a;
    }
}
  • Abstract class can not be declared as final

Understanding interfaces
Conceptually:
  • Interface is like a contract between two software components.
  • One component provides services according to this contract (called as service provider component) and the other component consumes services according to this contract (called as service consumer).

Syntactically:
  • Interface is a list of abstract functions.
  • The class implementing the interface MUST define ALL abstract functions that are present in the interface. 
  • If it fails to implement even one abstract function from the interface, the class itself becomes abstract and must be declared as abstract.
The Contract

interface IDataManager
{
    public void CompressData(Object obj);
    public void DeCompressData(Object obj);
    public int SaveData(Object obj);
    public abstract Object LoadData();
}
Writing a method abstract is optional. By default all methods of an interface are abstract.
  • IDataManager specified the terms of services for managing saving, loading, compressing and decompressing the data. 
  • FileDataManager provides services according to this contract using File system for storing data.
  • DatabseDataManager provides services according to this contract using Database system for storing data.

Important Points to note:
  • One class extend from ONLY one base class but can implement more than one interfaces.
  • One Interface say iA can be derived from another interface iB. A class implementing iA must define all methods of iA as well as iB.
  • One Interface say iA can be derived from more than one interfaces say iB and iC. A class implementing iA must define all methods of iA, iB as well as iC.​​​​​​​
Types of interfaces
There are three types of interfaces in Java.
  1. Normal interface
  2. Marker interface
  3. Functional interface
1. Normal interfaceThis type of interface can have more than one abstract methods

They are used for declaring a contract with the implementing class.
2. Marker interfaceThis type of interface don't have any abstract method. They are empty.

This is used when a class just shows interest in acquring an additional type. For doing that, it has to do nothing. 
3. Functional interfaceThis type of interface can have ONLY one abstract method.
This is mainly designed for lamda expressions
