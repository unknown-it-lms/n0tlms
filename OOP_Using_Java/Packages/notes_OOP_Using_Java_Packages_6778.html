Packages
  • In many situations, we need to use same class name for different classes that have all-together different purpose. For example, in a software system we may need to create reports for academic performance, financial performance as well as say, the performance of sales team.
  • For these three objectives, if we create a class Report three times, it will give error due to name collision.  
  • One solution is to create classes with different names like AcademicReport, FinancialReport and SalesReport. This may be very inconvenient if we want to create reports for few more departments in future.
  • A better solution is to create separate groups of related classes. All classes related to academics can be created in a group academics, all classes related to finance can be created in a group finance and all classes related to sales can be created in a group sales. These groups of related classes is called as packages.

Benefits of packages
Packages provide following two major benefits to us.
  1. They allow us to use same name for a class as long as it is in different package. 
  2. Due to logical grouping of similar classes, it is easy to develop and maintain the code in future also.
  3. We can declare a variable with package access specifier. Such variables can be accessed directly in any class in the SAME package. This way we can control the access of data in a better manner. Note that this is in fact, the default access specifier in Java.

This directory structure can be created in two ways.
1. ManuallyCreate the directory structure manually that matches the package structure. Copy and paste all .class files in appropriate directories 
2. With the help of Java compilerWe can simply ask the compiler to do all these tasks by specifying the destination of output, as follows.
> java -d . *.java-d specified the destination for creating the .class files in appropriate directory structure. Here . indicates current directory. So compiler creates the files and directory structure in current directory.

Appropriate directory structure will be created in the given folder and .class files are copied in appropriate directories by compiler automatically.
Setting classpath for classes created inside a package
> java -d . *.javaIf the package is compiled into current directory, we don't have to set the classpath because current directory is always in classpath by default. 
> java -d c:\know-it\bin *.javaIf the classes are compiled to other directory (c:\know-it\bin in this case), the the directory (c:\know-it\bin) must be added to the classpath so that we can access classes from that package.

This is generally done to arrange our classes in a more organized hierarchical structure. For example, inside the existing package Utility, we will now create sub-package Math, which contains a class Calculator inside it.

Package scope
  • Packages provide an additional access specifier. If we declare a class member with package access specifier, it can be accessed directly by all other classes present in the same package. 
  • This is the default access specifier, which means that if not specified explicitly, a class member is created with package as its access specifier. 
Class Date and Class Person are created inside same package Utility. 
package Utility;

public class Date {
    int d,m;
    private int y;
    // . . .
}		package Utility;

public class Person {
    String strName;
    Date dob;
  
    public Person(String str, int a,int b, int c) {
        strName=str;
        dob= new Date();
        dob.d=a; // OK
        dob.m=b; // OK
        dob.y=c; // ERROR
    }
  • No access specifier is mentioned for d and m for Date class. Hence, they have package scope.
  • So they can be accessed directly by any code written inside class Person
  • BUT data member y is explicitly declared as private. So it can't be accessed outside Date class even if it is in same package.
This same rule is applicable for classes also. If you want a class to be accessible outside its package, it must be declared as public class. If we don't make it public explicitly, it is created with package scope. It can't be accessed outside its own package in that case.   
jar files
If we want to distribute (share with client or other developers), all the classes created in different packages, we have to copy the entire directory structure on other machine. This is time consuming as well as it is quite error-prone. Java provides another easy method of distributing our classes by creating a single file that contains all the class files along with its directory structure. This file is called as a Java Archive file, in short a JAR file.

Benefits of using jar file:
  1. Distribution of classes is easy because we don't have to create directory structure on client machine
  2. Using the classes is also easy because we have to just add the entire jar file (along with its filename and ext), in the classpath on client machine.
  3. All classes are compressed and then stored. So the entire size of our distribution is less.   

Lets now see how to :
  1. Create a jar file
  2. See contents for a jar file
  3. Extract classes from the jar file
  4. Use classes from a jar file.
  5. Update jar file with new changes
  6. Create executable jar file and run it

1. Create a jar file:
Our source directory has Utility folder having Date.class and Person.class. We can compress this utility package into a jar file utility.jar as follows:

> jar cf utility.jar Utility
  • cf : create a new jar file
  • utility.jar: Name of new jar file
  • Utility: The folder whose content should be compressed recursively into the new jar file

4. Using classes from a jar file
To use classes from a jar file, we have to add the path along with filename and ext, in the classpath.
Once the jar file is added to classpath, we can import the packages inside it and start using the classes in our program.		Adding jar file to classpath from same folder (as the jar file)
C:\KNOW-IT\Java\bin>set classpath=%classpath%;./utility.jar

Adding jar file to classpath from any other folder
C:\KNOW-IT\Java> set classpath=%classpath%;C:\KNOW-IT\Java\bin\utility.jar
5. Update jar file with new changes
After creating the jar file, we may make some changes in the package (like adding some more sub-packages or editing existing classes etc). We must update our old jar file to include these changes. This can be done as follows:
> jar uf utility.jar		The new utility.jar file generated with this command is updated with all changes.
6. Make executable jar file and run it
The jar files created till now, can be referred by other java code. They can't be executed directly. If you want to make the jar file executable, you have to mention the class having main function in the manifest file. We want want to run the Test class using the executable jar file, we must mention it as the main class in manifest file sa follows.
Main-Class: Test
Now the jar file can be executed using following command:
> java -jar utility.jar
