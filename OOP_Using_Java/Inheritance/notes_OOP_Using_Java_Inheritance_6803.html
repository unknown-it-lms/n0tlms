Inheritance
  • Many times we need to create a more specialized form of an entity we already have in our system. For example, we already have an entity Person, modeled using class Person in our system.
  • This entity can be re-used for creating specialized types of persons like Employee. Note this is possible because every Employee is always a person first. It has all state and behavior of a person, to which, some more state and behavior is added to make it a more special type of person called as Employee.
  • Same is true for a Player also. The entity Person can be re-used for creating specialized types of persons like Player. Note this is possible because every Player is always a person first. It has all state and behavior of a person, to which, some more state and behavior is added to make it a more special type of person called as Player.
  • Inheritance makes such reuse possible in object oriented programming.
Concepts and Terminologies
1Inheritance is re-using existing entity (class) to create more specific forms of similar type.
2The class which is re-used is called base class (or Super class) and the new class created from it is called derived class (or sub class). Here, Person is base class (super class) and Employee is derived class (Sub class). Programmer is base class (super class) and DotNetProgrammer is derived class (Sub class).   
3We can further derive more specific classes from the derived classes. This way we create an hierarchy of classes which is called as inheritance hierarchy.
4Inheritance creates IS-A type of relationship between the two classes. This is because every derived type IS always of the base type. For example, every Player IS-A person and every Employee IS-A person.
5Classes at the top represent general entities and those at bottom represent more specific entities. For Example: Person is a general entity but Employee is more specific than Person. Manager is more specific entity than Employee and so on.
6​
7As we move down the hierarchy, we are moving from generalization to specialization.
8The derived class inherits entire state and entire behavior of the base class. It can only add more state and more behavior. Hence, everything we expect from base class can always be expected from the derived class.
9The IS-A relationship is applicable at single level as well as at multiple levels of inheritance. For example, every Employee IS-A Person, every Manager IS-A Employee. Hence, every Manager IS-A Person also. Similarly, we can say that every ProjectManager is Employee, every SalesManager is a Person, every DotNetProgrammer IS-A Person, every Programmer is Employee and so on... 
10As the derived class ALWAYS support entire state and behavior of base class, we can always provide an object of derived class where ever object of base class is needed. For example, If Person is needed for a task, it can be assigned to an Employee. If Programmer is needed for a task, we can provide JavaProgrammer. If Manager is needed for a task, we can always provide ProjectManager. This is called as Principle of Substitution, which say that "We can substitute objects of base class with any object of derived class in a software system".
11The IS-A relationship is unidirectional. Every SalesManager IS-A manager BUT every Manager may not be a SalesManager. Every Programmer IS-A Employee BUT every Employee may not be a Programmer. Hence, the arrow should point from sub class to super class.

class Employee extends Person {
    int empId; /* Employee adds empId and basicSal. This is modeled by adding new data members here */
    double basicSal;
    public Emp() { /* In the default constructor, we simply set the additional data members to zero. */
        empId=0;
        basicSal=0;
    }

    /*The parameterized constructor collects entire data needed for the super class as well as for its     additional data members.*/
    public Emp(String str, int a,int b, int c, int id,float sal) {

        /* The data needed for super class is passed to super class constructor using super(str, a, b, c) */
        super(str,a,b,c);
        
        /* The data which is needed by itself is used to initialize its own data members */ 
        empId=id;
        basicSal=sal;
    }
    public String display() {

        return super.display() + 
        " Id: " +empId+ /* Derived class only adds its own string representation */ 
        " Basic Sal: "+ basicSal;
    }

  1. Single Inheritance
  2. Multi-level Inheritance
  3. Hierarchical Inheritance
  4. Multiple Inheritance
  5. Hybrid Inheritance
 
1Single Inheritance
Code

Object class
In Java, if a class is not derived from any other class, it gets derived from a built-in class named "Object". This class is present in package java.lang. Hence, every class in Java is either directly or indirectly derived from this Object class. Following diagram shows this relationship.
​
KNOW-IT, Pune		Note that:
  • Classes like SalesManager, Employee are derived from some other class. Hence they are not directly derived from Object class. BUT class Person and Class Date are independent classes. They are not derived from any other class in our program.So they get derived from Object class automatically. 
  • Hence, Person and Date classes get derived from Object class directly and other classes like Admin, Employee, Programmer get derived from Object class indirectly.
  • This is applicable even for the built-in classes like String, Array etc
  • Object class is present in java.lang package.

Protected access specifier
We have already seen access specifiers like private, default and public. Inheritance adds a new access specifier called 'protected'. Any class member declared with protected access specifier can be accessed in any class derived from it directly or indirectly, even if the derived classes are in another package.  

Lets see its implementation as follows:

Person.javaEmployee.java

package utility;

public class Person {
    String strName;
    protected Date dob;
    // Other class members...
}		package staff;

import utility.*;
public class Employee extends Person {
    int empId;
    float basicSal;
   
    public String display() {
        String strName= this.strName; /* ERROR: strName is at package scope so it can't be accessed outside its own package utility. */
        Date dateOfBirth= this.dob; /*OK. dob is declared as protected hence it can be accessed in any class derived from Person even if it is in another package. */
        return dateOfBirth.display()+" Id: "+empId+" Basic Sal: "+ basicSal;
    }
}

class A {
    protected int a1;

    }

    }

    }

Note here that:
  • Protected members can be access directyl in any class derived directly or indirectly (in multiple levels of inheritance)

Review of all access specifiers in Java:
​​​​​​​Java has four access specifiers:
  1. private: Can be accessed only within the class in which it is declared
  2. default: Can be accessed from anywhere within the package. This is default specifier, if not mentioned explicitly.
  3. protected: Can be accessed from any class in the same package plus in any class derived from it directly or indirectly even when the child class is in another package.
  4. public: Can be access from anywhere, irrespective of package and class hierarchy.
