Fundamentals of OOP
In this chapter we will actually start understanding about object oriented programming.  We will learn advantages of object oriented programming, what is actual object, major and minor pillars of object oriented programming.
What is Procedural programming
 In procedural programming, large program is divided into smaller programs called procedures, which are also known as subroutines or functions. These functions operate on the program data called the variables. Each procedure contains a series of computational steps that will be carried out when the procedure is called.
In the procedure oriented programming approach, the main problem is divided into smaller tasks and the number of functions are written to accomplish these all tasks. These functions will have their own local variables and can even access global variables.

​​​​​​​​

Limitations of Procedural programming
Following are some of the limitataion of procedural programming.
  1. Basic unit of coding in a Procedural programming is a function, which models a task. Whereas in real life we understand a scenario by identifying various entities and interactions between them. Hence procedural programming does not model real-world entities like car, table, bank account, loan etc properly. 
  2. Procedural programming mainly focuses on procedures or functions. Less attention is given to the data. In fact data is the core part of any application. Data exists that's why application exists to manage and manipulate data. So it should be given more importance.
  3. Global data is directly accessible to all functions in a procedural program. If the global data somehow gets a wrong value, it is possible from any part of the code hence we have to debug the entire code. Thus, it becomes difficult for programmers to identify and fix issues.
  4. Limited and difficult code reusability.
  5. The procedural programming approach does not work well for large and complex systems.
What is Object Oriented Programming
​​​​​​​Object oriented programming treats data as a critical element in the program development and does not allow it to flow freely around the system. It ties data more closely to functions that operate on it, and protects it from accidental modification from outside function. Object oriented programming system allows decomposition of a problem into a number of entities called objects and then builds data and functions around these objects. The data of an object can be accessed only by the functions associated with that object.

Some of the features of object oriented programming are below:
  1. Emphasis is on data rather than procedure.
  2. Building block for the program is object.
  3. Functions that operate on the data of an object are tied together in one capsule.
  4. Data is hidden and cannot be accessed by external functions but can be only accessed by internal functions.
  5. Objects can communicate with each other through functions
  6. New functions and data can be easily added whenever need
  7. Object oriented approach follows bottom up approach in program design.
Advantages of OOP :
  1. OOP models the real life scenario better in a software system. It creates entities the interact with each other for doing some business.
  2. Entities provide better way of reusability then functions in procedural programming.
  3. Access to data can be controlled precisely only to few selected functions. Hence data is handled in more responsible way and hence more secured. In case the data gets a wrong value, we have only limited functions to debug. So code maintenance is easy.
  4. Interaction between different software systems is easy ( Due to interfaces and polymorphism, which we will see later) 
Concept of Class and Object
We have seen that OOP is better than procedural programming because it model the real life scenario better. Any real life scenario can be seen just as a collaboration between various entities. Objects in OOP models an these entities in real life.
Such entities can be tangible like a car, student, person etc or non-tangible like a bank account, a lecture, exam etc. All such entities should have following characteristics to be considered as object in a software system.
  1. State
  2. Identity
  3. Behavior
  4. Responsibility
Lets see each of them now:
State:
State is the set of values assigned to the attributes of an entity. 
For example: Consider a car as an object. Its attributes are Company, Model, Type, Price, Color, Current Speed etc. Every car in real life assigns some values to all these attributes, as given below for example.

  • Though all cars have different states, they share same set of attributes that describe them. Hence they belong to same category of entity which is called as Class. So all the above entities (car 1, car 2, car 3 etc) belong to same category (Car) but have different state.

  • A class is just a declaration of our design. So memory is not allocated when class is defined. It is only when object is created (in OOP term it is called instantiated), memory gets allocated.
Identity
We have seen that a class can have many attributes and objects assign different values to these attributes. It is possible that many objects assign same value to some of the attributes but there is at least one attribute which must have different value for different objects of that class. Such attribute is called as identity of that object. 
For example, in the case of class Car, there can be two or more cars having same color, same company, same model etc. So the attributes like color, Company, Model can't be considered as identity of car object. However, every car object MUST have different value assigned to the attribute "Registration Number". Hence, this attribute is called as the identity of car object.
Other examples of identity are: Roll Number for class Student, Account Number for class BankAccount, Transaction Id for class OnlineTransaction etc
Behavior
In real life, we use different objects by sending messages to them. For example, when we start a car, logically we sent a message "start" to the car object. When we stop a car, logically we send a message "stop" to the car object. Our car responds to these messages appropriately. This response of an object to external messages is called its behavior. 
We can see behavior of other objects as well for example.
"AttendClass",  "DoAssignments", "CompleteProject", "PayFees" are some of the behaviors of student object.
"ShowBalance", "TransferMoney", "PrintStatement" can be seen as some of the behaviors of bank account object. 
Responsibility
This is the overall role or purpose of an object in the given scenario. For example, the responsibility of bank account object is to maintain detail records of all transaction done from a given client account. Responsibility of a car object is to support all behavior so that the passengers can commute from one location to other.

For any entity (tangible or non-tangible), to be considered in a software system as object, it must fulfill all above conditions. It must have well defined state having at least one identity attribute. It must support all the behavior needed to fulfill its responsibility in the system.
Principles of OOP
We have previously seen the drawbacks of procedural programming. It doesn't model the real life scenario in software system properly. Object oriented programming achieves this by following some principles which are called as Principles of Object oriented programming. 
These principles are of major two types:
  • Major Principles 
  • Minor principles 

Major principles are applicable at the design level of the system and hence must be followed strictly. Following are the major principles of OOP.
  1. Abstraction
  2. Encapsulation
  3. Inheritance
  4. Polymorphism

Lets see details of each principle now.
Abstraction
The real life entity we want to model in our system has lot of attributes associated with it. However, only few of them are relevant for our software system. Abstraction is the process where we ignore all un-necessary attributes and consider only those which are needed for complete modeling of that entity in our system. Here, we select to ignore un-necessary details and hence in short it is also described as selective ignorance.
For example, if we want to model a student in our system, a student in real life has many attributes like name, date of birth, height, weight, blood group, academic details, place of birth, mobile number, email address and so on. However, for modeling a students in a College management system, we don't need height, weight, blood group etc. So we decide to ignore these attributes and only select to consider those which are meaningful for developing the college management system like name, date of birth, academic details, mobile number, email address etc. This is where we have done abstraction of student entity. 
Note that here we have done abstraction of state of object. Even the behavior expected from student entity is limited only to the state that we have selected to consider. For example, if we don't consider blood group of a student in our system, we won't ask the student what's his/her blood group. Hence abstraction of state automatically leads to abstraction of behavior. 
Encapsulation
In real life every entity has state and behavior associated with it. However, the state and behavior are closely coupled to each other. We can't separate the state from behavior in any case. For example, when we carry a mobile phone with us, we carry it's state (like name of company, color, dimensions, weight, camera details, SIM etc) as well as its behavior (like its ability to make call, receive call, send sms, receive sms, take picture etc).
Also note that in real life entities, not all details of an attribute are directly accessible or visible. For example, in mobile we just see the screen and keypad, not the internal electronic circuit. The SIM card is internally used by the mobile but not directly accessible to the user from outside. So a real life mobile not just binds state with behavior but also hides the sensitive and technical details of it from outside world and just makes the features available to them. 
This inseparable tight coupling of state with behavior and hiding sensitive technical details, for every object in real life is called encapsulation. Principle of encapsulation recommends the same to be done with the objects we model in our system.
Another example can be a car. It binds its state (Company, Model, Engine, Color, gear box, wheels etc) with its behavior (ability to start, stop, accelerate, change gear etc) to create a single logical capsule. It also hides the technical details like engine from direct access to the driver. 
Inheritance
In programming, it is recommended NOT to re-write a code if it is already written. Such reusability of code is possible even in procedural languages like C but the basic unit of reuse in such languages is a function. In object oriented languages, if some entity is already modeled by applying the principles of abstraction and encapsulation, the same can be re-used to create a more special types of it. This is recommended by the principle of inheritance in OOP.
Inheritance is re-using existing entity (class) to create more specific forms of similar type.
The class which is re-used is called base class and the new class created from it is called derived class.
Inheritance creates is-a type of relationship between the two classes. Classes at the top represent general entities and as we go down the hierarchy, the classes represent more specific entities. For Example: Person is a general entity but Employee is more specific than Person. Manager is more specific entity than Employee and so on.
​
Note the is-a relation created between the derived class entities with base class entities. Here every sales manger IS-A manager and every manager IS-A employee and every employee IS-A person. However, the reverse is not always true. For examples, not every person is employee, not every employee is manager and not every manager is sales manager. Same can be seen for other types. 
Polymorphism
In real life, it is observed in different scenarios that if we want to communicate with many entities of different types, instead of communicating with them individually, we just identify the common type between them. If the common type exists,we just send communicate with that common factor and indirectly it gets applicable to all the entities we want to communicate. 
Lets understand this better with an example. At a traffic signal, when the signal turns red, it actually wants to send a message "stop" to the bikes, cars, trucks, bicycles etc. But instead of sending different signals for all, it identifies the common factor between all these entities which is vehicle. So it sends the message "stop" to all vehicles and hence it gets automatically applicable to all vehicles like bikes, cars, trucks, bicycles etc. Upon receiving this common message, each of these vehicles do stop BUT they use different mechanism for implementation of this message. For example, bike may use disk-break mechanism, car may use hydraulic breaks, trucks may use air breaking system and bicycles may just use friction to stop it. Here all these objects follow the principle of polymorphism. 
Polymorphism is different implementation of same request by different objects which are related to each other through IS-A type of relation.
Note that the common message is targeted to the common factor (means the base class of all objects) and specific behavior is provided from the derived classes.

Lets now see the minor principles of OOP.

The minor elements can be absent in an object model But, if these elements are present in the model then it will be useful. But if we consider real life implementation, minor concepts are even equally important.
Strong typing : 
System needs a specific type of input to process with. Simple example - getting admission to PG course can be completed only by paying the fees using online mode of transaction. No other way of paying the fees like cash, cheque etc will be allowed by the system even though the amount in cash may be same.
Concurrency
Concurrency is the property by which a program can proceed from different fronts simultaneously. For example, is a car, the engine system, lubrication system, gear system, transmission system, music system all run at a time.
Persistence
Every object in the system takes some amount of space and exists for an amount of time.Persistency is the property by which an object can preserve its state permanently so that it can be retrieved back later. In a software system, this is specifically achieved by storing the state of object either on hard-disk through file input-output operations or in a database through database connectivity through program.
