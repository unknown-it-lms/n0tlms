JDBC
Java Database Connectivity(JDBC) is an Application Programming Interface(API) used to connect Java application with Database. JDBC is used to interact with various type of Database such as Oracle, MS Access, My SQL and SQL Server. JDBC can also be defined as the platform-independent interface between a relational database and Java programming. It allows java program to execute SQL statement and retrieve result from database.

Following are the concepts we need to understand in JDBC:
  • JDBC Drivers
  • Connecting to database
  • Executing Simple Queries
  • Prepared Statements
  • Callable Statements
  • Scrollable And Updatable Resultset
  • Batch Updates
  • Transactions
  • ​​​​​​​Reading metadata

Lets see each of them in details now.
JDBC Drivers
JDBC API does not directly communicate with the database. It uses JDBC driver of the database to interact with the database. JDBC driver is a software component provided along with the database which is required by the JDBC API to interact with the database. Each database will have its own JDBC driver.
In simple terms, JDBC drivers are nothing but the implementations of interfaces provided in the JDBC API (java.sql and javax.sql packages) for a particular database. These implementations are bundled in a JAR file and supplied along with the database. These implementations are used by the JDBC API to interact with that database.

JDBC drivers are of following four types: (Advantages are in green color, disadvantages in red color)
Type 1 JDBC Driver / JDBC-ODBC Bridge Driver
  • Provides the bridge between JDBC and ODBC API. It translates all JDBC calls into ODBC calls and sends them to ODBC driver which interacts with the database.
  • These types of drivers are slowest of all types. Because, all JDBC calls will go to the ODBC driver through the bridge and then to database. So it is time consuming and raises the performance issues. This type of drivers are not recommended for high transaction java applications.
  • This driver is not entirely written in java language, hence it causes portability issues.
Type 2 JDBC Driver
Translates all JDBC method calls into database specific calls using native API of the database.
Its performance is slightly better than the Type 1 driver as communication layer is reduced in this driver.
It is also not entirely written in java language. This causes the portability issues. This driver is database specific. So once you switch from one database to another, you have to change the driver.
Type 3 JDBC Drivers / Network Protocol Driver
It makes use of middle ware or application server that translates all JDBC calls into database specific calls.
Entirely written in java language. So no portability issues.
Extra application server or middle ware component has to be maintained.
Type 4 JDBC Drivers / Native Protocol Drivers
(Thin Driver)
It directly converts JDBC calls into database specific calls.
This driver is most popular among all 4 type of JDBC drivers.
It is preferred over Type 3 Driver as it removes extra layer of communication (Application Server / Middle ware) and this makes it faster than the Type 3 JDBC Driver.
Entirely written in java language and hence portable.

Connecting to databse
We need to execute following steps for connecting to database and run basic SQL queries related to CRUD operations. These steps are as follows:
  1. Register the Driver class
  2. Create connection
  3. Create statement
  4. Execute queries
  5. Close connection

1.Register the Driver class
The DriverManager class is the component of JDBC API and also a member of the java.sql package. It acts as an interface between the Java program and drivers. It contains all the appropriate methods to register and deregister the database driver class and to create a connection between a Java application and the database. 
Since JDBC 4.0, explicitly registering the driver is optional. We just need to put the Jar file in the classpath, and then JDBC driver manager can detect and load the driver automatically.

try
{
    Class.forName("com.mysql.cj.jdbc.Driver");
}
catch (ClassNotFoundException e)
{
    System.out.println("Class Not Found");
}
2. Create connection
For connecting to database, we need following information:
  • Connection URL: The connection URL for the mysql database is jdbc:mysql://localhost:3306/testdb where
    • jdbc is the API
    • mysql is the database
    • localhost is the server name on which mysql is running (we may also use IP address)
    • 3306 is the port number
    • testdb is the database name. 
  • Username: The default username for the mysql database is root.
  • Password: It is the password given by the user at the time of installing the mysql database. 

Connection con=DriverManager.getConnection(  "jdbc:mysql://localhost:3306/testdb","root","root");
3. Create statement
In this step, we create java.sql.Statement object. To create the Statement object, we use createStatement() method of Connection object obtained in step 2. There are different types of statements available based on the type of query to be executed in the database. Here we will focus on only simple statements without any parameters.

Statement stmt = con.createStatement();
4. Execute queries
We can send queries to the database using the Statement object obtained in step 3. This method returns one java.sql.ResultSet object which contains all the records returned by a query.
  • Use executeQuery() method for sending select queries to database (Select)
  • Use executeUpdate() method for sending DML statements to database (Create, insert, Update, Delete etc)

String sql = "select * from emp"; 
ResultSet rs = stmt.executeQuery(sql);
while(rs.next())
{
    //Process the records here
}

String sql = "delete from emp where empid=101"; 
int iRowsAffected = stmt.executeUpdate(sql);
5. Close connection
Once the connection is no more needed, we must close it as follows:

con.close();
Entire code that implements all the above steps is given below:

Prepared Statements
The PreparedStatement interface is a sub-interface of Statement. It is used to execute parameterized query. 
  • Use this when you plan to use the SQL statements many times.
  • The PreparedStatement interface accepts input parameters at runtime.
  • The PreparedStatement interface extends the Statement interface, which gives you added functionality with a couple of advantages over a generic Statement object.
  • This statement gives you the flexibility of supplying arguments dynamically.
  • The performance of the application will be faster if you use PreparedStatement interface because query is compiled only once.

 PreparedStatement pstmt=null;
try
{
    Class.forName("com.mysql.cj.jdbc.Driver");        
    Connection con=DriverManager.getConnection( "jdbc:mysql://localhost:3306/empdb","root","root");

    String strSql="insert into Emp values(?,?,?,?);";
    /* Note that sql query is passed as parameter for creating prepared statement object */
    pstmt=con.prepareStatement(strSql); 
    pstmt.setInt(1,103); //1 specifies the first parameter in the query which EmpId of type int
    pstmt.setString(2,"Anil"); //2 specifies the second parameter in the query which FirstName of type String
    pstmt.setString(3,"Roy"); //3 specifies the third parameter in the query which LastName of type String
    pstmt.setDouble(4,25300); //4 specifies the forth parameter in the query which BasicSalary of type double
  
​​​​​​​    int i= pstmt.executeUpdate();
            
    System.out.println(i+ " record inserted.");
    con.close();
}
catch(Exception ex) {
    System.out.println(ex.getMessage());
}
finally {
    pstmt.close(); /* Note that the PreparedStatement must be closed when not needed anymore. If you close the Connection object first, it will close the PreparedStatement object as well. However, you should always explicitly close the PreparedStatement object to ensure proper cleanup. */
}
Callable Statements
CallableStatement interface is used to call the stored procedures and functions. A stored procedure may have parameters of three types: IN, OUT, and INOUT.
The PreparedStatement object only uses the IN parameter. The CallableStatement object can use all the three.
  • IN: A parameter whose value is unknown when the SQL statement is created. You bind values to IN parameters with the setXXX() methods.
  • OUT: A parameter whose value is supplied by the SQL statement it returns. You retrieve values from the OUT parameters with the getXXX() methods.
  • INOUT: A parameter that provides both input and output values. You bind variables with the setXXX() methods and retrieve values with the getXXX() methods.

Scrollable And Updatable Resultset
By default, ResultSet object can be moved forward only and it is not updatable.
​​​​​​​
All methods which create a statement object, accept two parameters, which are:
  • int resultSetType: This parameter decides whether the result set object can scroll or not
  • int resultSetConcurrency: This parameter decides whether we can update a record using the resultset or not

Following are various values that can be assigned to these two parameter and their impact on the result object.
int resultSetType:
TYPE_FORWARD_ONLYThe result set is not scrollable (default).
TYPE_SCROLL_INSENSITIVEThe result set is scrollable but not sensitive to database changes.
TYPE_SCROLL_SENSITIVEThe result set is scrollable and sensitive to database changes.


int resultSetConcurrency:
CONCUR_READ_ONLYThe result set cannot be used to update the database (default).
CONCUR_UPDATABLE The result set can be used to update the database.

Following are various possibilities of creating different types of result set object:
Statement st1= connection.createStatement();
/*st1 is created without mentioning any values for resultSetType and resultSetConcurrency. Hence, default values will be considered for both parameters. Hence, the result set will be Forward only and read only*/
Statement statement = connection.createStatement(
ResultSet.TYPE_SCROLL_INSENSITIVE, ResultSet.CONCUR_UPDATABLE);
/*This will provide a scrollable and updatable result set*/
Statement statement = connection.createStatement(
ResultSet.TYPE_SCROLL_INSENSITIVE, ResultSet.CONCUR_READ_ONLY);
/*This will provide a scrollable and non-updatable (readonly) result set*/
Lets now see, how we can scroll the result set to any random position and update records using a scrollable updatable result set.
try
{
    Class.forName("com.mysql.cj.jdbc.Driver");        
    Connection con=DriverManager.getConnection("jdbc:mysql://localhost:3306/empdb","root","root");
    Statement stmt=con.createStatement(ResultSet.TYPE_SCROLL_SENSITIVE,ResultSet.CONCUR_UPDATABLE);
    String strSql="select * from emp";
    ResultSet rs= stmt.executeQuery(strSql);

    rs.absolute(2); // We can move to any position in the row directly as the resultset is scrollable

    /* We can update any column of current row directly as the resultset is updatable */

    rs.updateString(2,"Sunil"); // Column 2 gets a new value "Sunil"
    rs.updateString(3,"Joshi"); // Column 3 gets a new value "joshi"
    rs.updateRow();

    System.out.println(rs.getString(1)+" "+rs.getString(2)+" "+rs.getString(3));
    con.close();
}
catch(Exception ex) {
    System.out.println(ex.getMessage());
}
Batch Updates
Instead of executing a single query, we can execute a batch (group) of queries. It makes the performance fast. It is because when one sends multiple statements of SQL at once to the database, the communication overhead is reduced significantly, as one is not communicating with the database frequently, which in turn results to fast performance. The java.sql.Statement and java.sql.PreparedStatement interfaces provide methods for batch processing.
Batch update using Statement
try
{
    Class.forName("com.mysql.cj.jdbc.Driver");        
    Connection con=DriverManager.getConnection( "jdbc:mysql://localhost:3306/empdb","root","root");
    Statement stmt=con.createStatement();

    // Two records are added in the batch 
    stmt.addBatch("insert into emp values(201,'Abhinav','Jadhav',45000)");
    stmt.addBatch("insert into emp values(301,'Shrinivas','Jadhav',25000)");

    //Entire batch is executed now
    stmt.executeBatch();

    con.close();
}
catch(Exception ex) {
    System.out.println(ex.getMessage());
}
Batch update using PreparedStatement
try
{
    Class.forName("com.mysql.cj.jdbc.Driver");        
    Connection con=DriverManager.getConnection( "jdbc:mysql://localhost:3306/empdb","root","root");

    PreparedStatement pstmt=con.prepareStatement("insert into emp values(?,?,?,?)");

    pstmt.setInt(1, 401);
    pstmt.setString(2, "Ajay");
    pstmt.setString(3, "Sharma");
    pstmt.setDouble(4, 28500);
     pstmt.addBatch();

    pstmt.setInt(1, 501);
    pstmt.setString(2, "Suraj");
    pstmt.setString(3, "Varma");
    pstmt.setDouble(4, 38500);
    pstmt.addBatch();
    
    pstmt.executeBatch();
    con.close();
}
catch(Exception ex) {
    System.out.println(ex.getMessage());
}
Transactions
By default, the JDBC Connection is in auto-commit mode, which means that when we run the execute statements, changes are immediately done to the database. This may be suitable for most of the situations but there are some scenarios where we run more than one execute statements but we want that all of them should be successful together. If any one statement fails, we want to rollback even the previous statements which executed successfully. 

This can be achieved by taking following steps in JDBC:
  1. Set the auto-commit mode of connection object to false.
  2. Run multiple execute command
  3. If all execute successfully, commit the total changes to database.
  4. If any of the execute operations fails, rollback the previous execute operations also.
​​​​​​​
try
{
    Class.forName("com.mysql.cj.jdbc.Driver");        
    con=DriverManager.getConnection( "jdbc:mysql://localhost:3306/empdb","root","root");
    con.setAutoCommit(false);
    // Now run multiple execute operations which we want to be part of the transaction
    //...
    //...
    con.commit();
    con.close();
}
catch(Exception ex) {
    con.rollback();
    System.out.println(ex.getMessage()+" Transaction rolled-back");
}
Savepoint
Sometimes we want that if a transaction fails, instead of rolling back all actions, we just want to rollback actions only till a specific point. This point is called as savepoint.
We can create a save point and release it, if needed, using following methods:
  • setSavepoint(String savepointName) − Defines a new savepoint. It also returns a Savepoint object.
  • releaseSavepoint(Savepoint savepointName) − Deletes a savepoint. Notice that it requires a Savepoint object as a parameter. This object is usually a savepoint generated by the setSavepoint() method.
Transaction Isolation levels
JDBC provides support 5 transaction isolation levels through Connection interface.
  1. TRANSACTION_NONE: It is represented by integer value 0 does not support transactions.
  2. TRANSACTION_READ_COMMITTED: It is represented by integer value 2 supports transactions allowing Non-Repeatable Reads and, Phantom Reads.
  3. TRANSACTION_READ_UNCOMMITTED: It is represented by integer value 1 supports transactions allowing Dirty Reads, Non-Repeatable Reads and, Phantom Reads.
  4. TRANSACTION_REPEATABLE_READ: It is represented by integer value 4 supports transactions allowing only Phantom Reads.
  5. TRANSACTION_SERIALIZABLE: It is represented by integer value 8 supports transactions with out allowing Dirty Reads, Non-Repeatable Reads and, Phantom Reads.

We can set transaction isolation level as follows:
try
{
    Class.forName("com.mysql.cj.jdbc.Driver");        
    con=DriverManager.getConnection( "jdbc:mysql://localhost:3306/empdb","root","root");
    con.setAutoCommit(false);

    con.setTransactionIsolation(Connection.TRANSACTION_SERIALIZABLE);
    
    con.commit();
    con.close();
}
catch(Exception ex) {
    con.rollback();
    System.out.println(ex.getMessage()+" Transaction rolled-back");
}
Reading metadata
Metadata is information about some entity. In JDBC, we can find our more information about the database we are using as well as the result set returned by an execute statement.

This is provided by
  • ResultSetMetaData: Provides more information about Result set
  • DatabaseMetaData: Provides more information about the database

This is demonstrated in following examples:
Using ResultSetMetaData
try
{
    Class.forName("com.mysql.cj.jdbc.Driver");        
    Connection con=DriverManager.getConnection("jdbc:mysql://localhost:3306/empdb","root","root");
    Statement stmt=con.createStatement();
    String strSql="select * from emp";
    ResultSet rs= stmt.executeQuery(strSql);
    
    ResultSetMetaData rsmd= rs.getMetaData();

    int iCount= rsmd.getColumnCount();
    String strTableName= rsmd.getTableName(1);
    String strColumnName= rsmd.getColumnName(1);
    String strColumnType= rsmd.getColumnTypeName(1);
    
    System.out.println(iCount);
    System.out.println(strTableName);
    System.out.println(strColumnName);
    System.out.println(strColumnType);
    
    con.close();
}
catch(Exception ex) {
    System.out.println(ex.getMessage());
