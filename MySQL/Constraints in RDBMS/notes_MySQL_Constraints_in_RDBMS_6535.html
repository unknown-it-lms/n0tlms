Notes: MySQL- Constraints in RDBMS

Constraints in RDBMS      
Constraints  are the rules imposed on tables so that the data entered in the  table is a correct data.  Constraints are used to limit the type of data that can go into the table. This ensures the accuracy and reliability of the data in the table. If there is any violation between the constraint and the data action, the action is aborted.
​​​​​​​Any RDBMS gives us a facility to add, delete or update the data in the table. But many a times these operations need to be verified properly as there can be multiple dependencies on a single table. Hence constraints are used, so that the data though scattered in multiple tables remain consistent and integrated. Constraints are also called as data integrity constraints.      
    
​Constraints and their need
The database server uses constraints to prevent invalid data entry into tables. Constraints are used  to perform following tasks 
  •  Enforce rules on the data in a table whenever a row is inserted , updated, or deleted from that table.
  •  The constraint must be satisfied for the operation to succeed.
  •  Prevent the deletion/updation  of a table if there are dependencies from other  table.  ​​​​​​​

Types of constraints
RDBMS gives us the facility to create the constraints at either the column level or table level.
 Constraints in RDBMS are majorly classified into two types
  • Column Level Constraints 
        These constraints are applied only to the single column that limits the type of particular 
        column data.
  • Table Level Constraints 
        These constraints are applied to the entire table that limits the type of data for the whole table.

Data Integrity Constraints 
Following are the data integrity constraints used in RDBMS.
1. NOT NULL
If we specify a field in a table to be NOT NULL. Then the field will never accept null value. That is, you will not be allowed to insert a new row in the table without specifying any value to this field. This is a column level constraint.
2. UNIQUE    
This constraint helps in uniquely identifying  each row in the table. i.e. for a particular column, all the rows should have unique values. We can have more than one UNIQUE columns in a table. 
For example, the below query creates a tale Student where the field ID is specified as UNIQUE. i.e, no two students can have the same ID. Unique is a table level constraint.   
3. PRIMARY KEY
Primary Key is a field which uniquely identifies each row in the table. If a field in a table has primary key, then the field will not contain NULL values as well as all the rows will have unique values for this field. So, in other words we can say that this is combination of NOT NULL and UNIQUE constraints. A table can have only one primary key. Below query will create a table named Student and specifies the field ID as primary key.
4. FOREIGN KEY
Foreign Key is a field in a table which uniquely identifies each row of a another table. That is, this field points to primary key of another table. This usually creates a kind of link between the tables. Hence we can say that,
foreign key is a key, which is primary key in another table. Foreign key is a table lavel constraint. Foreign key is called as Referential Integrity. 
Referential integrety will ensure that the data in the table is consistant and integreated.
5. CHECK
Using the CHECK constraint we can specify a condition for a field, which should be satisfied at the time of entering values for this field. Check is a column level constraint.
6. DEFAULT
This constraint is used to provide a default value for the fields. That is, if at the time of entering new records in the table if the user does not specify any value for these fields then the default value will be assigned to them. This is column level constraint.

Using Constraints
Constraints can be implemented on the tables while table creation or we can implement constraints on already created table by using Alter Clause. Also we can delete existing constraints or change the constraints. Let us study this in details.
Adding constraint while creating table
 NOT NULL Constraint 
 If we specify a field in a table to be NOT NULL, then the field will never accept null value. That is, you will not be allowed to   insert a new row in the table without specifying any value to this field. This is a column level constraint. New versions of MySQL allow multiple nulls in a column.
 Syntax :   
 CREATE TABLE Student
       (
              ID int  NOT NULL,
              NAME varchar(10)  NOT NULL,
              ADDRESS varchar(20)
       );
Here, a table by name Student is created and a columns ID and NAME in the table will never accept null values while inserting the data in the table. In short my table will never have null values for ID and NAME columns .

2. UNIQUE Constraint
This constraint helps in uniquely identifying  each row in the table. i.e. for a particular column, all the rows should have unique values. We can have more than one UNIQUE columns in a table.
For example, the below query creates a table Student where the field ID is specified as UNIQUE. i.e, no two students can have the same ID.  
Syntax :
CREATE TABLE Student
      (
            ID int  NOT NULL UNIQUE,
            NAME varchar(10),
            ADDRESS varchar(20)   
      );
or
Applying Unique constraint with constraint name while table creation.
Syntax :
CREATE TABLE Persons
     (
            ID int UNIQUE,
            LastName varchar(20) NOT NULL,
            FirstName varchar(20),
            Age int,
            CONSTRAINT UC_Person_ID UNIQUE (ID)
        );
UC_Person is the constraint name.

3. Check Constraint
Using the CHECK constraint we can specify a condition for a field, which should be satisfied at the time of entering values for this field. Check is a column level constraint.
Syntax :
CREATE TABLE Persons
(
      ID int NOT NULL,
      LastName varchar(25 ) NOT NULL,
      FirstName varchar(25 ),
      Age int CHECK (Age>=18) ,
      City varchar(25)
 );
or
CHECK constraint with constraint name.
CREATE TABLE Persons
(
    ID int NOT NULL,
    LastName varchar(25) NOT NULL,
    FirstName varchar(25),
    Age int,
    City varchar(25),
    CONSTRAINT CHK_Person CHECK (Age>=18 )
);
CHK_Person is a constraint name.

4. Default Constraint

3. Composite Key
Composite key in SQL can be defined as a combination of multiple columns, and these columns are used to identify the rows  uniquely. Even though a single column can't identify any row uniquely, a combination of over one column can uniquely identify any record

The latest version of SQL supports the combining of more than 16 columns. The data type of the columns combined to make a composite key can be different for all the columns.

Altering constraints
Adding constraints to existing tables
1.  Adding NOT NULL constraint to an existing table Person
Syntax :
ALTER TABLE Person
MODIFY age int NOT NULL;
Here, on column age of table Person, NOT NULL constraint is applied after table creation. If column already contains null values, then it is not possible to apply NOT NULL conatraint.   

2. Adding CHECK constraint to an existing table Person
Syntax :
ALTER TABLE Persons
ADD CHECK (Age>=18);
or
ALTER TABLE Persons
ADD CONSTRAINT CHK_PersonAge CHECK (Age>=18);
Here, on column age of table Person, CHECK constraint is applied after table creation.If table has  values lesser than 18  for column age before the constraint is applied then it is not possible to apply a constraint.  

3. Adding DEFAULT constraint to an existing table Person
Syntax :
ALTER TABLE Persons
ADD DEFAULT city='PUNE';
or
ALTER TABLE Persons
ADD def_city DEFAULT city='PUNE';
Here, on column city of table Person, DEFAULT constraint is applied after table creation. In this case table may hold values for city as 'PUNE' if column city has null value while inserting a row. If table has null values before adding a constraint, then it is not possible to add a constraint.

4. Adding UNIQUE constraint to an existing table Person
Syntax :
ALTER TABLE Persons
ADD CONSTRAINT uni_empno UNIQUE (empno);
or
ALTER TABLE Persons
ADD  UNIQUE(empno);
Here, on column empno of table Persons, UNIQUE constraint is applied after table creation. In this case table will not hold duplicate values for column empno. If table already has duplicate values then UNIQUE constraint cannot be applied and it generates an error. 

5. Adding Primary Key constraint to existing table Person.
    Syntax:
         ALTER TABLE Persons
         ADD PRIMARY KEY (ID);
            or
         ALTER TABLE Persons
         ADD CONSTRAINT PK_Person PRIMARY KEY (ID );
         (with constraint name PK_Person)
    
6.  Adding Foreign Key constraint to existing table Person.
      Syntax:   
         ALTER TABLE Orders
         ADD FOREIGN KEY (PersonID) REFERENCES Persons(PersonID);
            or
         ALTER TABLE Orders
         ADD CONSTRAINT FK_PersonOrder
         FOREIGN KEY (PersonID) REFERENCES Persons(PersonID);

Deleting Constraints
 Many a times we need to make changes in database. The constraints specified earlier may not be required ,        and  changes need to be done accordingly. Hence SQL gives us the commands by which we can delete                  exsisting constraints of the table.
To delete existing constraint on the table, constraint should have a constraint name. The DROP CONSTRAINT command is used to delete a UNIQUE, PRIMARY KEY, FOREIGN KEY, or CHECK constraint.
Syntax :
ALTER TABLE  table_name
DROP CONSTRAINT constraint_name;
1. Droping check constraint. 
       ALTER TABLE Persons
       DROP CHECK CHK_PersonAge;
 Here, above command  will delete check  constraint.               
2. Droping unique constraint. 
       ALTER TABLE Persons
       DROP INDEX UC_Person;
 Here, above command  will delete unique  constraint.           
3. Droping Default constraints.
       ALTER City   
       DROP DEFAULT;           
4. Droping check constraint. 
         ALTER TABLE Persons
         DROP CHECK CHK_PersonAge;
Deleting Primary Key/ Foreign Key constraint
1. Droping primary key constraint.  
     ALTER TABLE Persons
     DROP primary key;
Here, above command will delete primary key constraint.
2. Droping foreign key constraint.
     ALTER TABLE Orders
     DROP FOREIGN KEY FK_PersonOrder;
Here, above command  will delete foreign key constraint.           
Summery
The constraint created can be deleted only if the constraint has a constraint name. So it is advisable to give the name for any constraint we create.  A constraint without a name cannot be deleted.
