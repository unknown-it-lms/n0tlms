Database Storage Structure
In a Relational Database Management System (RDBMS), data is stored in a structured format using tables, where each table consists of rows (records) and columns (attributes), same like a spreadsheet, with the key feature being the ability to establish relationships between different tables through shared values across them; this is known as the relational model.

Key points about RDBMS storage structure are
  •  Tables 
      The fundamental building block of an RDBMS, where data is organized into rows and columns. 
  • Rows (tuples/records) 
      Represent individual records within a table, containing values for each column. 
  • Columns (attributes) 
      Represent specific data fields within a table, each with a defined data type. 
  • Primary key:
       A unique identifier for each row within a table, used to establish relationships with other tables. 
  • Foreign key:
      A column in one table that references the primary key of another table, creating a link between them

Relational databases typically use tables with data organized into rows (containing entities) and columns (containing entity attributes). Deciding number of tables, it's attributes, and relationship between the tables is  done using  normalization. Each row holds a unique identifier or key that ties tables together to establish a relationship. When a relational database is queried, the key is used to find related data across the datasets.

Control File
A control file is a binary file that records the physical structure and configuration of a database. It contains information such as the database name, the location and size of the data files and redo log files, the checkpoint position, the database ID, and the backup history
​​​​​​​A control file is essential for opening and mounting a database, and for performing backup and recovery tasks
    
Control file plays a crucial role in DB recovery for below reasons 
1. It contains DB name, timestamp, unique DBID, required to identify and ensure integrity of DB during recovery.
2. It keeps track of tablespaces and data files like location of data files, size, online/offline status, with which it locates and validates data files.
3. It stores redo log file names, sizes, status, as they hold history of all changes made to DB to ensure correct redo log files are used during recovery.
4. It stores metadata of backup timestamps, locations, backup types to help select and restore proper backups during recovery.
5. It stores metadata of database schemas and data dictionary which would help to restore DB to a consistent state during recovery.

Tablespace  
Tablespace is a directory where data files are stored.  Oracle and MySQL databases use tablespaces and data files to store data. A tablespace is a logical storage unit, while a data file is a physical file that stores data for the tablespace.     
MySQL logical storage structure is similar to Oracle. It uses tablespaces for storing database objects, but the General Tablespace is not  supported. Only InnoDB file-per-table is provided.

innodb_file_per_table_n
The InnoDB file-per-table feature applies to each InnoDB table. Its indexes are stored in a separate .ibd data file. Each .ibd data file represents an individual tablespace.

File-per-Table Tablespace Advantages 
  • Space Management: Disk space is returned to the OS after truncating/dropping tables.
  • ALTER TABLE Efficiency: Avoids potential disk space bloat during table-copying ALTER TABLE operations.
  • TRUNCATE Performance: Generally faster TRUNCATE operations.
  • I/O Optimization: Allows data files to be placed on separate storage devices.
  • Table Import: Enables table import from other MySQL instances.
  • Row Format Support: Supports DYNAMIC and COMPRESSED row formats.
  • Recovery: Aids in data recovery scenarios.
  • Backup Flexibility: Allows for quicker backups/restores using tools like MySQL Enterprise Backup without affecting other tables.
  • Table Size Monitoring: Simplifies monitoring of table size through file system size.
  • O_DIRECT Performance: Potential performance gains when innodb_flush_method is set to O_DIRECT.
  • Size Limits: Each tablespace has a 64TB limit, enabling greater individual table sizes.
File-per-Table Tablespace Disadvantages 
  • Space Waste: Tables may have unused space that can only be used by rows of the same table.
  • fsync Overhead: More fsync operations are performed due to multiple data files.
  • File Handles: mysqld needs to maintain more open file handles.
  • File Descriptors: Requires more file descriptors.
  • Fragmentation: Potential for increased fragmentation that might impede certain operations, though this can be managed.
  • Buffer Pool Scan: Dropping tables can involve a potentially time-consuming buffer pool scan.
  • Auto-Extend Behavior: The innodb_autoextend_increment variable doesn't control file-per-table tablespace extensions; the extensions begin by small amounts and subsequently extend in 4MB increments.

innodb_system
The system tablespace is the storage area for the change buffer. It may also contain table and index data if tables are created in the system tablespace rather than file-per-table or general tablespaces.
The system tablespace can have one or more data files. By default, a single system tablespace data file, named ibdata1, is created in the data directory. The size and number of system tablespace data files is defined by the innodb_data_file_path startup option.
innodb_temporary
Non-compressed, user-created temporary tables and on-disk internal temporary tables are created in a shared temporary tablespace. The innodb_temp_data_file_path variable defines the relative path, name, size, and attributes for temporary tablespace data files. If no value is specified for innodb_temp_data_file_path, the default behavior is to create an auto-extending data file named ibtmp1 in the innodb_data_home_dir directory that is slightly larger than 12MB

Metadata tables
Data Dictionary tables are stored in the innodb_system tablespace.
A metadata table is defined as a table that contains information about which entities are generated as non-temporal tables and which entities are generated as asserted version tables in a logical data model. It helps in managing business keys and identifying objects represented in the database
.ibd files  
When it comes to MySQL databases, .ibd files are a fundamental aspect that plays a crucial role in maintaining data integrity and performance. These files are specifically associated with the InnoDB storage engine, which is widely recognized for its efficiency in handling data retrieval, indexing, and transaction management. The InnoDB engine utilizes a sophisticated combination of B-tree structures and row-level locking mechanisms to ensure not only data consistency but also effective concurrency control.
 
Structure of an .ibd File
The structure of an .ibd file comprises two primary elements: tablespace data and index data. These components work in tandem to facilitate seamless data management within MySQL databases.

Importance of .ibd Files
For MySQL databases, recognizing the significance of .ibd files is paramount to ensuring robust data management practices that prioritize both integrity and performance.
Tablespace Data
The tablespace data contained within an .ibd file serves as a dedicated storage area for table-specific information. This segment includes critical details such as field values, row identifiers, and other essential attributes that define the structure of the table.
Index Data   
On the other hand, index data plays a pivotal role in optimizing query performance by enabling rapid access to specific records within the database. By organizing key values efficiently, MySQL can expedite search operations and enhance overall system responsiveness.
Data Integrity
Maintaining data integrity is a core principle in database management, and .ibd files play a vital role in upholding this standard by safeguarding against corruption or loss of critical information.
Performance Optimization   
In addition to preserving data integrity, leveraging .ibd files can significantly enhance performance optimization within MySQL databases. By streamlining access to table-specific tablespaces and index structures, these files contribute to faster query execution times and improved overall system efficiency.
